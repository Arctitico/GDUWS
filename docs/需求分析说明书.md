# 需求分析说明书

## 一、引言

### 1.1 编写目的
本文档用于明确 GDUWS 的功能性需求与非功能性需求，指导开发团队理解系统应实现的功能和性能目标。

### 1.2 项目背景
随着无人装备在现代军事中的重要性日益增加，开发一款模拟无人装备战争的游戏不仅具有娱乐价值，还能为相关领域的研究提供一个实验平台。GDUWS 旨在通过高仿真的战场环境和复杂的对抗机制，探索无人装备在各种战术情境下的表现和潜力。

### 1.3 适用范围
本项目仅包含单机双阵营对抗推演，不包含联网对战、资源采集建造、单位升级树、Mod系统。

### 1.4 术语定义与缩写

| 术语 | 定义 |
|------|------|
| `GDUWS` | Ghost Domains: Unmanned Warfare Sim |
| `UC` | Use Case（用例），描述用户与系统之间的一次完整交互，从用户视角回答"用户能用系统做什么" |
| `FR` | Functional Requirement（功能性需求），描述系统必须具备的功能特性，回答"系统要做什么" |
| `tick` | 游戏逻辑帧，1 秒 = 30 tick，所有战斗时序（冷却、超时、移动）均以 tick 为最小单位 |
| 布兵阶段（DEPLOY） | 战斗开始前，玩家在地图上放置己方单位的阶段 |
| 结算（RESULT） | 战斗结束后展示胜负结果与统计数据的阶段 |
| 阵营（Faction） | 当前仅有 PLAYER（玩家）与 ENEMY（敌方）两个阵营 |
| 配额 | 当前布兵机制下，每种单位类型有固定数量上限，放置消耗、移除返还 |
| 禁布区 | 地图上标记为不可部署单位的区域，布兵阶段以暗色迷雾蒙版渲染 |
| 侦察（SCOUT）/ 打击（STRIKE） | 单位的两种任务角色。侦察单位负责探索地图、发现敌人；打击单位负责追击并摧毁已知敌人 |
| IDLE | 打击单位在无敌情可攻击时的待命状态，闲置超过 90 tick（约 3 秒）后自动转为侦察 |
| 攻击域 | 单位可命中的目标高度层集合，分为对地（Land）、对水面（WaterSurface）、对空（Air）、对水下（Underwater） |
| 情报板（IntelBoard） | 每个阵营维护一张共享敌情表，侦察单位发现的敌人写入该表，打击单位据此选择攻击目标 |
| 战争迷雾 | 玩家视角受己方单位视野范围限制，未在视野内的区域暗色渲染、敌方单位不可见 |
| 视野范围（sightRange） | 单位可探测敌方的圆形半径（像素），决定战争迷雾的可见区域和侦察单位的情报采集能力 |
| 攻击范围（maxAttackRange） | 单位可发起攻击的最大圆形半径（像素），选中单位后以橙色实线圈渲染 |
| 僵持（stalemate） | 双方超过 2500 tick（约 83 秒）无任何单位损失时触发，按双方损失率判定胜负 |

### 1.5 状态标记说明
每条功能性需求均标注实现状态：

| 标记 | 含义 |
|------|------|
| ✅ | 已实现 |
| 🔲 | 未实现 |
| 🚧 | 部分实现 |

### 1.6 参考项目
本项目参考 **RustedWarfare（铁锈战争）** 开发。

## 二、用户角色

### 2.1 玩家
操作员，负责选择关卡、布置兵力、设置策略、观察战斗过程并查看结果。

### 2.2 用例总览

| UC-02 | 选择关卡 | 从关卡列表中选择一个关卡 | ✅ |
| UC-03 | 部署单位 | 在地图可通行格上放置己方单位 | ✅ |
| UC-04 | 切换角色 | 左键点击已部署单位，切换其侦察/打击角色 | ✅ |

| UC-09 | 重新挑战 | 重新挑战当前关卡 | ✅ |

## 三、功能性需求

### 3.1 启动与选关

| 编号 | 需求描述 | 验收标准 | 状态 |
|------|---------|---------|------|

| FR-03 | 点击关卡进入布兵阶段 | 地图正确渲染，可正常布兵 | ✅ |
| FR-04 | 可以通过地图编辑器自定义关卡 | 可自定义关卡并保存到本地，可正常游玩 | 🔲 |

### 3.2 布兵阶段

| 编号 | 需求描述 | 验收标准 | 状态 |
|------|---------|---------|------|
| FR-05 | 侧栏显示关卡可用单位类型及剩余配额，配额随放置/移除实时更新 | 数量正确增减 | ✅ |
| FR-06 | 左键在可通行格放置选中单位；不可放置时拒绝并提示 | 校验正确，提示清晰 | ✅ |
| FR-07 | 左键点击己方单位切换侦察/打击角色 | 角色切换后侧栏有文字反馈 | ✅ |
| FR-08 | 右键点击己方单位将其移除 | 单位消失，侧栏数量恢复 | ✅ |
| FR-09 | 侧栏可切换放置的单位类型和默认角色 | 后续放置使用新设置 | ✅ |
| FR-10 | 禁布区以暗色迷雾蒙版显示，仅在布兵阶段可见（与 FR-20 战争迷雾统一色系） | 蒙版渲染正确，禁布区与非禁布区区隔清晰 | ✅ |
| FR-11 | 布兵阶段玩家不可见敌方预置单位（依赖 FR-20 战争迷雾） | 布兵阶段地图上不渲染敌方单位；战斗开始后由己方视野揭示 | ✅ |
| FR-12 | 兵种按价格计费，关卡给予固定资金（`playerFunds`），玩家在资金范围内不限数量自由组合兵力（`playerBudget` 此时仅作可用兵种名册，数值不再构成数量上限）；放置/移除实时扣减/返还资金，买不起的兵种置灰。关卡未给资金时退化为 FR-05 的固定配额制（向后兼容） | 每种单位有明确价格；侧栏显示剩余资金；资金不足时拒绝放置并提示；不再叠加数量上限 | ✅ |
| FR-13 | 兵种简介面板：玩家可在布兵阶段点击"兵种简介"查看每种可用兵种的详细属性（移动域、生命、视野、攻击域、射程、伤害、价格等） | 弹窗显示完整兵种信息，格式清晰 | ✅ |

### 3.3 战斗阶段

| 编号 | 需求描述 | 验收标准 | 状态 |
|------|---------|---------|------|
| FR-14 | 点击"开始战斗"后所有单位进入自动推演，玩家不可操控 | 布兵按钮禁用，单位按 AI 自主行动 | ✅ |

| FR-18 | 打击单位周边敌方兵力超过友方 2 倍时自动撤退 | 撤至安全距离，兵力改善后重新投入 | ✅ |

| FR-23 | 玩家可在战斗阶段通过侧栏实时调整 AI 总体策略倾向 | 侧栏提供策略控件 | 🔲 |

## 四、非功能性需求

每条非功能性需求统一编号（NFR-xx），供《测试设计及结果报告书》追溯。

| 编号 | 类别 | 需求描述 | 验收标准 |
|------|------|---------|---------|
| NFR-01 | 性能 | 战斗阶段以 30 tick/s 稳定推进，单 tick 处理 ≤ 33ms，运行时无用户可感知卡顿 | 战斗过程帧率稳定，无明显掉帧 |
| NFR-02 | 性能 | 50+ 单位场景下缩放/平移视口仍保持 ≥ 30 FPS | 大规模单位下交互流畅 |
| NFR-03 | 可用性 | 侧栏、按钮、字体、配色等视觉元素协调统一 | 界面风格一致、清晰易懂 |
| NFR-04 | 健壮性 | 配置文件缺失或格式错误时给出明确错误提示并拒绝启动，不崩溃、不静默跳过 | 异常输入有清晰报错，程序不崩溃 |
| NFR-05 | 部署 | 发行包自包含，双击 `GDUWS.exe` 免装 Java 直接启动 | 无 JDK 环境亦可运行 |
| NFR-06 | 可用性 | 键鼠操作符合常规 RTS 习惯；操作失败时侧栏给出文字提示 | 操作直观，失败有反馈 |
| NFR-07 | 可用性 | 结算界面明确展示胜负、判定原因与双方损失统计 | 结果信息完整明确 |
| NFR-08 | 可靠性 | 完整自动战斗推演数千 tick 全程不抛异常、不中断 | 长时间推演稳定 |
| NFR-09 | 可扩展性 | 新增单位类型仅需添加 JSON 文件、新增关卡仅需添加 JSON + `.map` 文件，均不改 Java 代码 | 数据驱动扩展生效 |
| NFR-10 | 可维护性 | 主源码零第三方依赖（仅音频解码用 JOrbis/JOGG；JUnit 仅测试期引入，不进发行包） | 编译不依赖外部库 |
| NFR-11 | 兼容性 | 仅支持 Windows，构建需 JDK 17+（含 `jlink`/`jmods`） | 在目标平台正常构建运行 |
| NFR-12 | 内容质量 | 关卡敌方兵力、地形与禁布区配置应使侦察-打击协同、撤退等核心玩法得到充分体现 | 关卡设计能体现核心机制 |

## 五、业务规则（BR）

业务规则是对功能性需求中关键判定逻辑的细化，提取自代码实现中的可测试约束，供《测试设计及结果报告书》逐条追溯。括号内为关联 FR 及关键常量（详见《概要设计说明书》《详细设计说明书》）。

### 5.1 布兵规则

| 编号 | 规则描述 | 关联 |
|------|---------|------|
| BR-02-1 | 每种单位有固定配额，放置消耗 1、移除返还 1；配额耗尽后再放置无效 | FR-05、FR-08 |
| BR-02-2 | 禁布区、与单位 `movementType` 不符的地形、以及与已有单位同格重叠时，均拒绝放置 | FR-06 |
| BR-02-3 | 单位仅能部署/移动于其 `movementType` 允许通行的地形（地形通行矩阵见《概要设计说明书》§5.3.3） | FR-06 |
| BR-02-4 | 新部署单位的默认角色为 STRIKE（打击） | FR-07、FR-09 |

### 5.2 战斗与 AI 规则

| 编号 | 规则描述 | 关联 |
|------|---------|------|
| BR-03-1 | 视野内敌方写入本阵营 `IntelBoard`；情报超过 `INTEL_MEMORY_TIMEOUT = 300` tick 未更新则过期剔除 | FR-16、FR-20 |
| BR-03-2 | 仅 `sightRange` 内的敌方被目击；友方不计入；双方各自独立记录情报 | FR-20 |
| BR-03-3 | A* 路径仅经过该 `movementType` 可通行格；起点即终点返回空，终点不可达/越界返回 null | FR-15 |
| BR-03-4 | 侦察避战（`avoidEnemies=true`）时，对已知敌人 `THREAT_RANGE = 6` 格内叠加威胁代价，路径自然绕开 | FR-15 |
| BR-03-5 | 战斗选射程内最近且攻击域可命中的目标开火并进入冷却；单 tick 转向不超过 π/16；移动沿 `path` 按 `moveSpeed` 推进 | FR-17、FR-21 |
| BR-03-6 | 打击单位连续 IDLE 满 `STRIKE_IDLE_TIMEOUT = 90` tick 自动转侦察；情报板出现敌人后立即转回打击 | FR-19 |
| BR-03-7 | 半径 `THREAT_RADIUS` 内 友方 HP < 敌方 HP × `RETREAT_RATIO(0.5)`（即敌方 > 友方 2 倍）时，向已知敌人质心反方向撤退 | FR-18 |

### 5.3 结算规则

| 编号 | 规则描述 | 关联 |
|------|---------|------|
| BR-04-1 | 某方 `lossRatio > 0.9`（存活 < 10%，严格大于）即判该方失败；恰好 90% 损失不触发 | FR-24 |
| BR-04-2 | 双方 `STALEMATE_TIMEOUT = 2500` tick 无任何兵力损失触发僵持判定；任一损失事件重置计时 | FR-25 |
| BR-04-3 | 僵持判定按损失率裁决：损失率高者负，两方相等时玩家胜 | FR-25 |

## 六、外部接口约定

### 6.1 配置文件体系
- 单位属性、关卡设定、地图均通过外部配置文件定义，格式为 JSON 和纯文本
- 新增单位类型或关卡仅需添加配置文件，无需修改 Java 代码
- 具体字段定义与格式说明见《概要设计说明书》数据设计章节

### 6.2 数据目录结构
- 单位配置：`data/units/*.json`
- 关卡配置：`data/levels/*.json`
- 地图文件：`data/maps/*.map`

### 6.3 异常处理
- 配置文件缺失或格式错误时，程序应给出明确的错误提示并拒绝启动，不应崩溃或静默跳过
- 地图文件行数不足、关卡引用了不存在的单位 ID 等异常情况同理
