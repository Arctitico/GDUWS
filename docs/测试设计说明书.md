# 测试设计说明书

> **项目名称**：Ghost Domains: Unmanned Warfare Sim（GDUWS）
> **文档性质**：单元测试与集成测试设计（含可运行的 JUnit 5 测试代码）
> **配套需求**：《需求分析说明书 v2.0》
> **日期**：2026-06-06

---

## 1. 文档目的与范围

本文档面向 GDUWS 的**核心业务逻辑**设计单元测试与集成测试，将《需求分析说明书》中的功能性需求（FR）、业务规则（BR）、非功能性需求（NFR）与数据约束（§5）转化为可自动执行、可追溯的测试用例，并给出执行结果与发现的偏差。

- **纳入自动化测试**：模型层（`model`）、数据层（`data`）、控制层（`control`）——均为纯逻辑、无 GUI/线程依赖，确定性强、可重复。
- **不纳入自动化测试**（见 §10）：视图层（Swing 渲染与鼠标交互）、音频层（后台播放线程）、主循环计时（`GameLoop`），这类需求以**手工/探索性测试**覆盖。

---

## 2. 测试策略

| 维度 | 选择 | 说明 |
| ---- | ---- | ---- |
| 测试框架 | **JUnit 5（Jupiter）** | 业界标准；`junit-platform-console-standalone` 单 jar，无需 Maven/Gradle |
| 构建方式 | `test.ps1`（PowerShell + `javac`） | 与现有 `compile.ps1`/`run.ps1` 一致，零构建工具 |
| 依赖约束 | 仅**测试期**引入 JUnit | 主源码仍满足 NFR-10「零外部依赖（除 JOrbis/JOGG）」；JUnit 不进入发行包 |
| 测试分层 | 单元测试为主、集成测试为辅 | 符合测试金字塔：底层快、广；集成验证跨模块链路 |
| 隔离手段 | 纯代码夹具 `testkit.Fixtures` | 不依赖磁盘即可构造地图/单位/世界；数据驱动集成测试才读真实 `data/` |
| 命名约定 | 单元测试 `*Test`，集成测试 `*IT` | `*IT`（Integration Test）与单元测试区分；运行时用 `--include-classname` 一并纳入 |

**测试设计方法**：等价类划分、边界值分析（如 90% 损失边界、情报记忆超时边界、转向速率上限）、判定表（攻击域矩阵、地形通行矩阵）、状态转换（打击/侦察 FSM）、Given-When-Then（直接复用需求验收标准）。

## 3. 测试环境与运行方式

| 项目 | 值 |
| ---- | ---- |
| JDK | 17+（实测 Temurin/Microsoft OpenJDK 21） |
| 测试框架 | JUnit 5 / Platform Console Standalone `1.11.4`（位于 `code/libs/`） |
| 主源码目录 | `code/src/main/java` → 编译至 `code/out/` |
| 测试源码目录 | `code/src/test/java` → 编译至 `code/out-test/` |
| 运行脚本 | `code/test.ps1` |

**运行全部测试**：

```powershell
cd GDUWS/code
.\test.ps1
```

**只运行某个测试类**：

```powershell
.\test.ps1 com.gduws.model.PathfinderTest
```

脚本流程：编译主源码 → 编译测试源码 → 用 JUnit Console 扫描 `out-test/` 执行（过滤器同时匹配 `*Test` 与 `*IT`）。

## 9. 发现的偏差与缺陷

测试以**规格为基线**编写，发现以下**实现/数据与规格的偏差**。均已在测试中显式记录，由项目方决定是改实现/数据还是修订规格。

### 缺陷 D-1：90% 损失判定在 10% 边界与 FR-04 不一致（建议修复实现）

- **现象**：`World.checkVictory` 使用 `lossRatio >= 0.9` 判负。当存活率**恰好 10%**（如 2/20）时 `lossRatio = 0.9`，触发判负。
- **规格**：BR-04-1 / FR-04 验收明确「存活数 / 初始数 **< 10%**」（严格小于）才判负，故 10% **不应**触发（验收用例：「玩家方存活 2 / 初始 20…存活率 = 10%，不触发」）。
- **证据**：`WorldVictoryTest.loss90_atExactly10pct_currentBehavior_characterization`（特征化当前行为，通过）+ `loss90_atExactly10pct_perSpec_FR04`（按规格断言，`@Disabled`，记录期望）。
- **建议**：将 `World.java` 中 `lossRatio >= 0.9` 改为 `lossRatio > 0.9`，随后启用被禁用的规格测试。

### 缺陷 D-2：战列舰/驱逐舰攻击域数据与 §5.1.2 矩阵不一致（建议核对数据或规格）

- **现象**：`data/units/battleship.json` 与 `destroyer.json` 的 `canAttackLand = true`。
- **规格**：§5.1.2 攻击域矩阵中「战列舰」「驱逐舰」的「陆地」列为 ✗（不可打陆）。
- **证据**：`DataDrivenLoadIT.attackDomainsCharacterizeDataVsSpec`（特征化数据现状，通过并注明偏差）。
- **建议**：二者取其一——若规格为准则把数据 `canAttackLand` 改为 `false`；若设计已变更则更新 §5.1.2 矩阵。

### 备注 N-1：JUnit 默认扫描过滤器不匹配 `*IT`（已在测试基建中处理）

- JUnit Console 默认仅扫描 `*Test`/`*Tests` 命名的类；集成测试以 `*IT` 命名会被漏扫。
- `test.ps1` 已通过 `--include-classname` 追加 `*IT` 模式修正，确保 8 个集成测试被纳入执行。

## 10. 未覆盖范围与后续建议

以下需求**不宜**或**难以**用 JUnit 自动化（涉及 Swing 渲染、鼠标交互、音频线程、实时计时），建议以**手工/探索性测试**覆盖：

| 需求 | 手工测试要点 |
| ---- | ---- |
| FR-05 战场观察 | 左键选中显示攻击范围圆与寻路；框选多选；滚轮缩放；右键拖动平移 |
| FR-06 背景音乐 | 菜单播 `starting/`、战斗切 `gaming/`；音乐缺失时静默不报错 |
| NFR-01 主循环帧率 | 战斗阶段 30 tick/s，单 tick ≤ 33ms（可加性能埋点） |
| NFR-02 视口渲染 | 50+ 单位缩放/平移 ≥ 30 FPS |
| NFR-05 零安装启动 | 双击 `GDUWS.exe` 免装 Java 直接启动 |
| NFR-06 操作直观 | 布兵侧栏：单位选择/角色切换/开始按钮/兵力统计清晰 |
| NFR-07 结果明确 | 结算界面展示胜负、判定原因、双方损失统计 |

**后续建议**：

1. 修复 D-1（一字符改动）后启用 `loss90_atExactly10pct_perSpec_FR04`，使 90% 边界规格被测试守护。
