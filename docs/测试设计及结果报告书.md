## 一、引言

### 1.1 编写目的

本文档面向 GDUWS 的**核心业务逻辑**设计单元测试、集成测试与可视化场景测试，将《需求分析说明书》中的功能性需求（FR）、业务规则（BR）、非功能性需求（NFR）与数据约束（§5）转化为可自动执行、可追溯的测试用例，并给出执行结果与发现的偏差，作为质量验收与回归测试的依据。

### 1.2 参考文档

- 《需求分析说明书》（FR-01~FR-38、BR-02-1~BR-04-3、NFR-01~NFR-12、§五 业务规则、§六 外部接口约定）
- 《概要设计说明书》（§三 分层架构、§5.4 单位属性总表、§7.3 攻击域矩阵、§5.3.3 地形通行矩阵、§5.3 文件格式约定）

> **需求编号对照**：本文档早期版本使用粗粒度功能分组编号（FR-01 关卡选择 / FR-02 兵力部署 / FR-03 自动战斗 / FR-04 胜负判定），现已统一对齐《需求分析说明书》现行编号：关卡选择→FR-02/FR-03，兵力部署→FR-05~FR-09，自动战斗→FR-14~FR-21，胜负判定→FR-24~FR-26；细粒度判定逻辑统一追溯到业务规则 BR-xx。FR-21（射弹）两版含义一致。

### 1.3 测试范围

- **纳入自动化测试**：模型层（`model`）、数据层（`data`）、控制层（`control`）——均为纯逻辑、无 GUI/线程依赖，确定性强、可重复。
- **不纳入自动化测试**（见 §十一）：视图层（Swing 渲染与鼠标交互）、音频层（后台播放线程）、主循环计时（`GameLoop`），这类需求以**手工/探索性测试**覆盖。
- **可视化场景测试**：复用游戏渲染链路、以与集成测试相同的确定性场景做演示与离屏自检，断言验证仍以无界面 JUnit 套件为准。

### 1.4 状态标记说明

| 标记 | 含义 |
|------|------|
| ✅ | 已覆盖 / 已通过 / 已修复 |
| ⚠️ | 存在偏差，待项目方决策 |
| 🔲 | 未自动化（手工测试覆盖） |

## 二、测试策略

### 2.1 测试分层

```
┌─────────────────────────────────────────────┐
│ 可视化场景测试 (Scenario)  复用渲染链路演示与离屏自检 │
├─────────────────────────────────────────────┤
│ 集成测试 (Integration, *IT)  验证跨模块链路与完整推演 │
├─────────────────────────────────────────────┤
│ 单元测试 (Unit, *Test)  验证各模块独立功能          │
├─────────────────────────────────────────────┤
│ 测试夹具 (testkit.Fixtures)  纯代码构造地图/单位/世界 │
└─────────────────────────────────────────────┘
```

### 2.2 测试方法与约束

| 维度 | 选择 | 说明 |
| ---- | ---- | ---- |
| 测试框架 | **JUnit 5（Jupiter）** | 业界标准；`junit-platform-console-standalone` 单 jar，无需 Maven/Gradle |
| 构建方式 | `test.ps1`（PowerShell + `javac`） | 与现有 `compile.ps1`/`run.ps1` 一致，零构建工具 |
| 依赖约束 | 仅**测试期**引入 JUnit | 主源码仍满足 NFR-10「零外部依赖（除 JOrbis/JOGG）」；JUnit 不进入发行包 |
| 命名约定 | 单元测试 `*Test`、集成测试 `*IT` | 运行时用 `--include-classname` 一并纳入；方法名遵循 `方法_条件_预期` |
| 隔离手段 | 纯代码夹具 `testkit.Fixtures` | 不依赖磁盘即可构造地图/单位/世界；仅数据驱动集成测试读真实 `data/` |

**测试设计方法**：等价类划分、边界值分析（如 90% 损失边界、情报记忆超时边界、转向速率上限）、判定表（攻击域矩阵、地形通行矩阵）、状态转换测试（打击/侦察 FSM）、Given-When-Then（直接复用需求验收标准）。

## 三、测试环境与运行方式

### 3.1 软件环境

| 项目 | 值 |
| ---- | ---- |
| 操作系统 | Windows 11 |
| JDK | 17+（实测 Temurin / Microsoft OpenJDK 21） |
| 测试框架 | JUnit 5 / Platform Console Standalone `1.11.4`（位于 `code/libs/`） |
| 主源码目录 | `code/src/main/java` → 编译至 `code/out/` |
| 测试源码目录 | `code/src/test/java` → 编译至 `code/out-test/` |
| 测试数据 | `code/data/`（单位、地图、关卡） |
| 运行脚本 | `code/test.ps1` |

### 3.2 运行方式

```powershell
# 运行全部测试
cd GDUWS/code
.\test.ps1

# 只运行某个测试类
.\test.ps1 -Select com.gduws.model.WorldVictoryTest

# 可视化回放（手动观察 AI 行为与战斗推演）
.\test.ps1 -Visual
```

脚本流程：编译主源码 → 编译测试源码 →（默认）用 JUnit Console 扫描 `out-test/` 执行并在终端输出（过滤器同时匹配 `*Test` 与 `*IT`）；（`-Visual`）启动 `ScenarioViewer` 渲染窗口。

### 3.3 可视化场景回放器（可选演示模式）

`testkit/ScenarioViewer` 复用游戏自身的 `GameRenderer`/`GamePanel`，用**与集成测试相同的确定性场景**构造 `World`，按 30 tick/s 逐帧渲染，让人直接看到单位移动、寻路、开火、射弹与胜负判定；底部状态栏实时显示与断言对应的状态（情报共享、双方存活、胜者）。

| 项目 | 说明 |
| ---- | ---- |
| 交互 | 上一个/下一个场景、重新开始、暂停、0.5x~4x 调速；鼠标滚轮缩放、右键拖动平移 |
| 自检 | `java ... com.gduws.testkit.ScenarioViewer --selftest` 离屏渲染各场景，校验渲染链路不崩（无显示器/CI 可用） |
| 定位 | 仅用于「演示/观察」，**断言验证仍以无界面 JUnit 套件为准** |

## 十、发现的偏差与缺陷

测试以**规格为基线**编写，以下偏差已在测试中显式记录，由项目方决定是改实现/数据还是修订规格。

### D-1：90% 损失判定 10% 边界 —— ✅ 已修复

- **原现象**：`World.checkVictory` 曾使用 `lossRatio >= 0.9` 判负，存活率**恰好 10%**（如 2/20）时 `lossRatio = 0.9` 会误触发判负。
- **规格**：BR-04-1 / FR-24 明确「存活数 / 初始数 **< 10%**」（严格小于）才判负。
- **当前实现**：`World.java` 已改为 `if (lossRatio > 0.9)`，恰好 10% 不再触发，`WorldVictoryTest` 中对应规格用例（TC-W-002）已转为正式断言并通过。

### D-2：战列舰/驱逐舰攻击域 —— ✅ 已确认一致

- **结论**：战列舰、驱逐舰**可对地打击**。数据文件 `battleship.json` / `destroyer.json` 的 `canAttackLand = true`，与《概要设计说明书》§7.3 攻击域克制矩阵（二者「打地」列为 ✓）一致，无偏差。
- **背景**：早期文档曾以过时的 §5.1.2 矩阵（陆地列为 ✗）为基线，将该项记为偏差；现矩阵与数据已统一，相关断言由特征化测试改为正向校验。
- **证据**：`DataDrivenLoadIT.attackDomainsMatchSpec` 正向断言二者 `canAttackLand` 为 true，通过。
